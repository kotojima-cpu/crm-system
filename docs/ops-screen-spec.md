# 運営管理画面 仕様書

## 1. 画面一覧

すべて `/platform/*` パスに配置し、`platform_admin` ロールのみアクセス可能。

| # | 画面ID | 画面名 | URL | 概要 |
|---|--------|--------|-----|------|
| 1 | OPS-001 | ダッシュボード | /platform | テナント概要・KPI |
| 2 | OPS-010 | テナント一覧 | /platform/tenants | テナント検索・一覧 |
| 3 | OPS-011 | テナント登録 | /platform/tenants/new | 新規テナント作成 |
| 4 | OPS-012 | テナント詳細 | /platform/tenants/:id | テナント情報・利用状況 |
| 5 | OPS-013 | テナント編集 | /platform/tenants/:id/edit | テナント情報編集 |
| 6 | OPS-020 | 請求一覧 | /platform/billing | 全テナントの請求管理 |
| 7 | OPS-021 | 請求詳細 | /platform/billing/:id | 請求詳細・手動調整 |
| 8 | OPS-030 | お知らせ一覧 | /platform/announcements | お知らせ管理 |
| 9 | OPS-031 | お知らせ作成 | /platform/announcements/new | お知らせ作成・配信 |
| 10 | OPS-040 | 利用レポート | /platform/reports | テナント別利用統計 |
| 11 | OPS-050 | 監査ログ | /platform/audit-logs | 全テナントの監査ログ閲覧 |

```
│                                                              │
│  ── 注意が必要なテナント ──                                     │
│  [!] テナント「GHI」: 顧客数がsoftLimitの90%に到達（450/500件） │
│  [!] テナント「JKL」: hardLimit到達（150/150件）- 登録不可       │
│  [!] テナント「MNO」: 2月分請求が未払い（overdue）              │

```

**データソース:**
- テナント数: `COUNT(tenants)`
- 今月売上: `SUM(invoices.total_with_tax) WHERE billing_period = 当月 AND status IN ('confirmed','sent','paid')`
- softLimit到達テナント: `顧客数 / softLimit >= 0.9` のテナント
- hardLimit到達テナント: `顧客数 >= hardLimit` のテナント
- 未払い請求: `invoices WHERE status = 'overdue'`

### 3.5 OPS-013: テナント編集

OPS-011（テナント登録）と同一レイアウト。

**差異:**
- 初期管理者ユーザー作成セクションなし

- ステータス変更はテナント詳細画面の停止/再開ボタンで行う
- customerLimit, softLimit, hardLimit は編集可能

**データモデル:**

```prisma
model Announcement {
  id            Int       @id @default(autoincrement())
  title         String    @db.VarChar(200)
  body          String
  targetType    String    @map("target_type") @db.VarChar(20)  // "all" | "plan" | "specific"
  targetPlan    String?   @map("target_plan") @db.VarChar(50)
  importance    String    @default("normal") @db.VarChar(20)   // "normal" | "important" | "urgent"
  scheduledAt   DateTime? @map("scheduled_at")
  publishedAt   DateTime? @map("published_at")
  status        String    @default("draft") @db.VarChar(20)    // "draft" | "scheduled" | "published"
  createdBy     Int       @map("created_by")
  createdAt     DateTime  @default(now()) @map("created_at")
  updatedAt     DateTime  @updatedAt @map("updated_at")

  creator       User      @relation(fields: [createdBy], references: [id])
  tenantTargets TenantAnnouncement[]

  @@index([status, scheduledAt], map: "idx_announcements_status_scheduled")
  @@map("announcements")
}

model TenantAnnouncement {
  id              Int       @id @default(autoincrement())
  tenantId        Int       @map("tenant_id")
  announcementId  Int       @map("announcement_id")
  readAt          DateTime? @map("read_at")
  createdAt       DateTime  @default(now()) @map("created_at")

  tenant        Tenant       @relation(fields: [tenantId], references: [id])
  announcement  Announcement @relation(fields: [announcementId], references: [id])

  @@unique([tenantId, announcementId])
  @@map("tenant_announcements")
}
```

**配信方式:** ポーリング方式。テナントユーザーのヘッダーに未読お知らせバッジを表示。
画面ロード時に `GET /api/announcements?unread=true` で未読件数を取得。

## 4. platform 管理画面共通仕様

### 4.1 操作対象テナントの明示

テナント個別データを表示する全ページ（テナント詳細、請求詳細等）では、画面上部に以下を常時表示する:

```
操作対象テナント: {テナント名}
```

### 4.2 テナント横断操作の監査

platform_admin のテナント横断操作（テナントデータの閲覧・編集・停止等）は全て監査対象である。
操作時に `execution_context: "platform"` と `targetTenantId` が監査ログに記録される。

### 4.3 顧客数表示の文言ルール

全画面で統一する表現: **「バッチ実行時点の顧客数」**

**禁止表現（使用不可）:**
- 「請求時点の顧客数」
- 「月末時点の顧客数」
- 「前月末時点の顧客数」

**理由:** 実装上はバッチ実行時点（毎月1日 00:30）のスナップショットであり、
厳密な月末時点ではないため、誤解を防ぐ。

### 4.4 hardLimit 到達時の表示文言

| 表示先 | 文言 |
|-------|------|
| tenant 側（顧客登録画面） | 「顧客登録数が上限（{hardLimit}件）に達しています。プランのアップグレードについては管理者にお問い合わせください」 |
| platform 側（ダッシュボード・テナント詳細） | 「テナント「{name}」の顧客数が hardLimit（{hardLimit}件）に到達 — 新規登録不可」 |
